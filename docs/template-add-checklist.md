# テンプレート追加実装 チェックリスト（標準手順）

新しいワーク種別テンプレートをアプリに追加するときの標準手順。
S-M系5種（S_G12/S_G38/S_G78/S_M12/S_M15、PR #35）の実装経験をもとに標準化した。

うまく働いた3つの要因を毎回再現することが目的:

1. **参考テンプレの実物指定** — 「G18_62のようにして」のように既存テンプレを指すと、言葉での説明より速く正確に伝わる
2. **実装前Q&A** — 設計判断（工程の統合・削除、ブロックの差し替え等）をコードを書く前に全部確定させる。実装してからバグを探すのではなく、実装前にバグを潰す
3. **ゴールデンテスト** — 「既存ワーク種別の出力に差分ゼロ」を機械的に証明しながら結線変更を入れる

---

## 全体フロー

```
フェーズ0  ドラフト作成（開発者・機械側の知見で .nc を書く）
フェーズ1  計画    — 仕様確定シートを埋める・疑義リストを潰す
フェーズ2  設計    — 参考テンプレを決め、変換後の抜粋と生成例で事前レビュー
フェーズ3  実装    — テンプレ変換＋registerWorkType登録＋結線（標準は gui-v2.html の script 追加のみ）
フェーズ4  確認    — 静的チェック → ゴールデン追加 → 目視 → PR → マージ
フェーズ5  改善    — 詰まった点を本チェックリストに追記して次回へつなぐ
```

順番の原則: **数値・仕様の確定（1）→ 構造の確定（2）→ 機械作業（3）→ 証明（4）** の順に進める。
後工程で前工程の決定を覆すと手戻りが大きいため、フェーズ1・2で「もう聞くことがない」状態を作ってから実装に入る。

---

## フェーズ0: ドラフト作成の推奨規約

ドラフト（`テンプレート/*.nc`）は確認中の草稿としてAIの変更対象外（`.js` 化の指示があるまで触らない）。
書く側が以下を守ると、フェーズ1の質問とフェーズ3の変換ミスが大きく減る:

- 可能なら仮マーカーではなく**本物のプレースホルダー**で書く（`X{{最大径-5}}` 等）。
  仮マーカー（`X##.` + `(SAIDAIKEI-5.0)` のような注記）を使う場合は、意味が一意に読めるコメントを必ず添える
- 冒頭コメントに仕様値を書いておくと質問が減る（例: `(SPEC: ID=10.0 DRILL=7.0 FLAT_TOOL=8)`）
- 確定済みの表記規約（S-M系実装時に確定）:
  - ヘッダーは2行形式: `O{{入力_工程No}}(PM-{{入力_図番}}=No,{{入力_工程No}}=Q)` ＋ `(S-G12)`（版表記はプログラム内のみ。ファイル名/workType ID/UIラベルには付けない）
  - コメントにスラッシュを使わない（`G1/2` → `G12`）
  - 工具プレースホルダーは**行の一番右端**（例: `G0G40G97S750M3{{外径荒}}`）
  - スーパー系工程の M458/M459 はコメント化 `({{M458}})` で統一
  - F値は `F.04` 形式（`F0.04` にしない）
  - Nシーケンス番号は重複・欠番なしの連番

## フェーズ1: 計画 — 仕様確定シート

実装前に以下を**すべて**確定させる（1つでも未確定のまま実装に入らない）。
ドラフトから推定できる値も、必ず数値を提示して承認をもらう。

| 項目                                                    | 反映先                                                            | 例（S_G12）                                             |
| ------------------------------------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------- |
| workType ID / ファイル名 / UIラベル / グループ / 表示順 | テンプレJS `registerWorkType(ui)`                                 | `S_G12` / `data_template_S_G12.js` / S-G12 / スーパー系 |
| 内径加工径D                                             | テンプレJS `registerWorkType(machining.idDiameterMm)`             | 10.0                                                    |
| ドリル径                                                | テンプレJS `registerWorkType(machining.drillDiameterMm)`          | 7.0                                                     |
| 刃長制限の有無                                          | テンプレJS `registerWorkType(machining.drillMaxDepthMm)`          | なし                                                    |
| ドリルモード                                            | logic-v2.js ドリルブロック選択チェーン                            | G74/G1 ユーザー選択                                     |
| 平底工具径（平底許可時）                                | テンプレJS `registerWorkType(machining.flatBottomToolDiameterMm)` | 8                                                       |
| 許可する加工スタイル                                    | テンプレJS `registerWorkType(ui.styles)`                          | 平底/通常/ヨセ/ヨセ中継/交差穴小                        |
| M99P100                                                 | UI 2択 / M40専用x50u8 / 非表示                                    | 2択                                                     |
| 角あり（W√2計算）対応                                   | `{{最大径+角}}` `{{最大径+3}}` の結線                             | 対応する                                                |
| NCL015もみつけ要否                                      | テンプレ/logic                                                    | 不要                                                    |
| **使用工具キーの全機械充足**                            | data-v2.js                                                        | `{{スーパー}}` を全機械に追加                           |
| 参考にする既存テンプレ                                  | フェーズ2の土台                                                   | template_G18_62                                         |

**工具キー充足チェック**は `npm run check:machine-tools`（`scripts/check-machine-tools.mjs`、`npm run check` に含む）が
自動で行う。テンプレが使う `{{機械キー}}` が全機種の機械定義に存在するかを検証し、未定義があればエラーにする
（S-M系実装時、`{{スーパー}}` が NCL044 にしか定義されておらず他機械で未解決になるところだった事例を機械的に防ぐ）。
無効化したい機種は値を空 `""` にする（存在すれば OK。空は設備差による意図的設定として正常扱い）。

あわせて**疑義リスト**を作る。ドラフト内の表記ゆれ・N番号の重複や欠番・既存テンプレと食い違う
非標準パターン（例: S_M12 の NEJI-MEN が既存M12と逆順だった件）は、1件ずつ「直すか・意図か」を
確認してから実装する。

## フェーズ2: 設計 — 変換方針の事前レビュー

- 参考テンプレを1つ決め、**差分だけ**を議論する（全部を言葉で説明しない）
- 判断が分かれそうな箇所（鬼門）は、**変換後テンプレの抜粋＋生成されるGコードの実例**を
  通常モード・角ありモードの両方で提示してから確定する。プレースホルダー名だけの説明は
  誤解の元（S-M系実装時、`{{最大径+角}}`/`{{最大径+3}}` の使い分けはこの方法で確定した）
- 確定済みの構造ルール:
  - **外径仕上げのN2統合**（G18_62方式）: 仕上げパスは `N100` ブロック内に置く。M99P100 の
    ON/OFF どちらの経路でも外径仕上げまで N2 内で完結し、別シーケンスの工具アプローチ往復を省く
  - **外径仕上げ3行の挿入位置と座標**: ノーズR終端の逃げ行（輪郭終端X+0.2）の直後に挿入。
    1行目 `{{最大径+角}}Z-◯F.3` の Z＝**ノーズR開始のZ**、2行目 `Z-◯F.03` の Z＝**ノーズR終端（製品全長）**、
    3行目 `{{最大径+3}}X◯F.08{{M459}}(-X-)` の X＝**輪郭終端X+0.1**。端面さらいは3行の後
  - N3 の内径深さ行は `Z-{{入力_内径深さ}}F.05(-Z-){{ヨセパス}}`、次行に `{{平底_内径仕上出口}}`、
    ブロック末尾の後に `{{ヨセブロック}}(YOSE Blocks)`（既存Mねじ系と同形）
  - `(M99P100)` 行は `(M99P100){{M99P100}}` に変換

## フェーズ3: 実装 — 結線チェックリスト

テンプレ本体:

- [ ] `テンプレート/data_template_<ID>.js` を作成（`const template_<ID>=\`…\`;`）。ドラフト `.nc` は削除（git上はrename）
- [ ] フェーズ0の表記規約とフェーズ2の構造ルールをすべて適用
- [ ] ファイル末尾に `registerWorkType({ id, ui, machining, features, template })` を追加。**UI表示（ラベル/グループ/表示順/許可スタイル）・加工径3種・刃長制限・MH/Tube属性はすべてここに書く**。`logic-v2.js` の径マップ・`gui-v2.js` の UI 定義は登録情報から自動生成されるので直接編集しない。詳細は CLAUDE.md「ワーク種別レジストリ」節を参照

アプリ結線（フェーズ1〜5移行済み。**標準ワーク種別はこの2つだけ**）:

- [ ] `gui-v2.html` — `<script>` タグ追加
- [ ] `assets/data-v2.js` — 不足している工具キーを**全機械**に追加（新工具が必要な場合のみ）

特殊挙動が必要な場合のみ（後段フェーズ未移行のため個別分岐が残る）:

- [ ] `assets/logic-v2.js` — Tube のチューブデータ置換・M40 の x50u8・G12B のノーズR・M12系バリアント選択のような**特殊処理**を伴う場合のみ、`registerWorkType(behavior)` に名前を付け `WORK_TYPE_BEHAVIORS` に `applyXxxBehavior()` を追加（標準テンプレ選択はレジストリ経由なので不要）
- [ ] `assets/blocks-v2.js` — 平底出口など特殊挙動が必要な場合のみ

確認:

- [ ] `npm run check:worktypes` で径マップ整合とレジストリ整合を確認

## フェーズ4: 確認

順番どおりに実行する:

1. [ ] `npm run check`（プレースホルダー整合・参照ファイル・lint・全テストを含む）
2. [ ] ゴールデンケース追加: **Normal＋許可スタイル各1件＋角あり系1件**を最低ラインとする
3. [ ] **標準テスト条件**を使う（`docs/golden-tests.md` 参照）:
   - 通常: 母材幅A=B=33.5 → 外径最大径 **47.376**（=√(A²+B²)）
   - 角あり: W=33.5 / H=11 → 外径最大径 **64.83**（=√(((W/2+H)×2)²+W²)）
   - **ブリッジルール**: UIが自動計算する値（外径最大径など）は手で適当な数値を書かず、
     UIと同じ計算式（`computeMaxOdResult()`）で算出した値をケースに書く。
     捏造すると実UIで起こり得ない入力になり、偽の異常出力を検証することになる
     （S-M系実装時、角ありケースで maxOD を捏造し `{{最大径+角}}` のXが `{{最大径+3}}` の
     Xより小さい偽出力を作ってしまった）
4. [ ] `npm run test:golden:update` → 新規スナップショットを**目視レビュー**
   - 検算ポイント: 最大径系3プレースホルダーの数値、内径深さ・ドリル深さ、平底出口行、
     ヨセ/ヨセ中継/交差穴の自動計算値（手計算と突き合わせ）
   - 角ありでは「`{{最大径+角}}` 行のXが後続 `{{最大径+3}}` 行のXより小さくならない」ことを確認
5. [ ] **既存スナップショットに差分がないこと**（あれば意図の説明が必要。CLAUDE.md のルール参照）
6. [ ] PR作成 → スナップショット抜粋をユーザーに提示 → **確認が取れてからマージ**
       （Gコード出力に関わる変更のため、CLAUDE.md の「マージ前にユーザーに確認」に該当）

## フェーズ5: 改善

- 実装中に詰まった点・新たに確定した規約・新しい鬼門は、**その場で本チェックリストに追記**する
  （次回のフェーズ0〜2の質問を減らすことが目的）
- バグを修正したら `scripts/golden/cases/regression/` にケースを1件追加する
  （`docs/golden-tests.md` の運用ルール）
