# ドリル加工ロジック抽出（logic-v2.js / blocks-v2.js / gui-v2.js より）

> スコープ: **ドリル加工そのものの分岐ルールだけ**を対象にする。
> 内径バイト（ボーリングバー）による平底仕上げ・奥バイト面取りなど「内径加工バイトが前提」の加工方法
> （例: M40 / M22 / M18 / M15 / G78・M42X3系・トメセン系・G12B・M12のNormal/バイト仕上げ）は、
> ドリルは単なる下穴でありユーザーがG74/G1を選べて特に矛盾がないため、本ドキュメントでは扱わない。
>
> 経緯: UIには常に「ドリルモード（G74/G1）」の選択肢が出るが、一部ワーク種別ではロジック側で
> その選択が無視されて固定動作になっていた。この違和感を解消するため、2026-07-18にUI側で
> 「選択できるケース／固定のケース」を出し分ける改修を実施した（gui-v2.js）。さらに現場確認により
> **M12のHGDR選択時もG1固定であるべき**という実装漏れ（バグ）が判明したため、これも同日中に修正した。
> 本ドキュメントはその根拠になっているコード分岐を整理したもの。
>
> 抽出元:
> - `Gコードジェネレータ/assets/logic-v2.js`（ドリルブロック選択チェーン・ドリル深さ計算）
> - `Gコードジェネレータ/assets/blocks-v2.js`（Gコードブロック本体）
> - `Gコードジェネレータ/assets/gui-v2.js`（UI側の固定判定・プロファイル解決・刃長制限警告）
> - `Gコードジェネレータ/テンプレート/data_template_*.js`（実際の使用工具の裏付け）

## 1. ドリルの分類は「ワーク種別」ではなく「物理的な工具」で決まる

テンプレートを確認した結果、G74/G1固定・選択の別は work種別そのものではなく、**どの物理ドリル工具を使うか**で決まっている（工具が同じなら挙動も同じ）ことが分かっている。

| 工具 | 挙動 | 理由 | 対象 |
|---|---|---|---|
| `{{ドリルHGDR}}`（HGDR） | **G1単動固定**。ペック・ステップ一切なし。送り F.15固定 | HGDR工具はペック/ステップが不要な設計 | G18全種（4系=仕上げ／6系=下穴、役割は違うが工具もロジックも同じ）／M12のHGDR選択時 |
| `{{ドリルHSS}}`（HSS）・ASWD系工具 | **10mmステップ送り固定**。送り F.1 | ねじれのあるHSSドリルはペック/ステップが必要 | M8_21・M8_31（HSS系）／J_M8_300・J_M8_200（次世代ASWD系）／M12のHSS選択時 |
| `{{ドリルT}}`（普通の下穴ドリル） | **G74/G1をユーザーが選択可能** | 内径バイト前提の下穴。G74なら送りF.25、G1ならF.15 | M12のBAITO（通常バイト加工）／M40等の内径加工バイト前提ワーク全般 |

## 2. ドリルブロック選択チェーン（`logic-v2.js` ドリルブロック選択チェーン）

```
① usesG18DrillShiageG1Block(workType) または isM12HgdrFinish(workType, m12FinishType)
                                                                 → G1固定（単動）
   対象: G18_40 / G18_42 / G18_62 / G18_655 / G18_6175（+MH全て）、M12/M12_MH×HGDR

② isJM8ASWDWorkType(workType) または isM8WorkType(workType)     → 10mmステップ送り固定
   対象: J_M8_300 / J_M8_200（次世代ASWD系）、M8_21 / M8_31（HSS系）

③ (workType===M12||M12_MH) かつ m12FinishType==="hss"           → 10mmステップ送り固定

④ それ以外（M12のNormal/バイト含む、内径加工バイト前提のワーク全般）
                                                                 → ユーザーが選んだ drillMode（G74/G1）が有効
```

`isM12HgdrFinish()` は `m12FinishType` が `"hss"`/`"baito"` 以外（`"hgdr"`／`"halfmoon"` のどちらの表記でも）なら真を返す。この2つの文字列表記が混在している理由は、Ichimonjiスタイル用UIカードの値が直接 `"hgdr"` を渡すのに対し、CrossSmall/YoseRelay用は歴史的に `"halfmoon"` という内部名にマッピングされているため（テンプレート選択の三項演算子も同様に「`hss`でも`baito`でもなければHGDR」というフォールバックで両方を吸収している）。

### ①のドリルブロック中身: `getDrillShiageHGDRBlock(depth, "G1")`（`blocks-v2.js`）

単動1発切削のみ。`G1Z-depth F.15` → `G4U.3`（ドウェル） → `Z1.F2.5`（引き戻し）。ペックサイクル（G74）は使わない。

### ②③のドリルブロック中身: `getDrillShiage10mmStepBlock(depth)`（`blocks-v2.js`）

G1単動固定、10mm刻みで「切削→Z30.まで引き戻し→次ステップ手前まで早送り」を繰り返し、最終深さのみドウェル(`G4U.3`)を入れて終了。**G74は一切使わない。ステップ量は常に10mm固定（ドリル径によって変わらない、確認済み）。**

参考: ④のグループが使う `getDrillShiageHGDRBlock` のG74モードは、深さ30mm以下なら一発切込み、30mm超なら「最初30mmをG74（Q3000）→以降10mmステップ」という深さ依存の分岐がある。①②③にはこの深さ分岐は存在しない（常に同じ動き）。

## 3. 一文字ドリル（Ichimonji）との関係

内径スタイル `Ichimonji`（一文字DR平底）を選ぶと、上記①〜④のドリルブロックとは別に、専用工具「一文字ドリル」を使うブロックが追加で走る（`logic-v2.js`）。

| 条件 | 使用ブロック | 内容 |
|---|---|---|
| style="Ichimonji" かつ (M12/M12_MH または M8系) | `getIchimonjiHirazokoBlock(内径深さ)` | 内径深さ基準（Depth-2.0〜Depth+0.2）で平底仕上げ |
| style="Ichimonji" かつ 上記以外の全ワーク | `getIchimonjiBlock(CP)` | CP±2mmで貫通させる面取り |

- M12/M8のIchimonjiは、①〜③のメインドリル（下穴）＋一文字ドリルによる仕上げ、の2段構成。
- M8系は選択肢なし固定（`m8Profile` は常に `"drill_ichi_men"`）。

## 4. ドリル径・刃長制限（刃の物理的な長さによる加工深さ上限）

`DRILL_DIA_MAP`（`logic-v2.js`）と、現場確認済みの刃長制限（`gui-v2.js` `DRILL_MAX_DEPTH_MM` / `getDrillMaxDepthMM()`）:

| ワーク種別 | ドリル径 (mm) | 刃長制限（加工可能深さ上限） |
|---|---|---|
| G18_40 / _MH | 4.05 | 54mm |
| G18_42 / _MH | 4.15 | 54mm |
| G18_62 / _MH | 4.15 | 54mm |
| G18_655 / _MH | 4.15 | 54mm |
| G18_6175 / _MH | 4.15 | 54mm |
| M12 / M12_MH（HGDR選択時のみ） | 4.05 | 54mm（G18と同一物理ドリルのため） |
| M8_21 | 2.1（ワーク寸法公差 2.1±0.2） | 24mm |
| M8_31 | 3.0（ワーク寸法公差 3.1±0.2） | 30mm |
| J_M8_300 | 3.0（ワーク寸法公差 3±0.1） | 30mm |
| J_M8_200 | 2.0（ワーク寸法公差 2±0.1） | 24mm |

※ 実際のドリル加工径は公称径より約0.05mm小さくなる（現場からの指摘により2026-07-18訂正: M8_21は2.2→2.1、M8_31は3.2→3.0に修正）。

※ φ4系ASWD（40mmまで）は対応するワーク種別が現状システムに存在しないため未実装（今回は無視）。

刃長制限を超える深さが指定された場合、**ブラウザのポップアップ警告を出すのみ**（`window.alert`）。生成そのものは止めない＝そのまま進めることも、値を修正して戻ることも自由（`gui-v2.js` の `next-depths` アクション内でチェック）。

ASWDドリル肩高さ `ASWD_SHOULDER_MM`（ドリル公称径キー）: `4.0→0.96` / `3.0→0.70` / `2.1→0.50` / `2.0→0.50`

自動計算式（`logic-v2.js`、ドリル関連のみ抜粋）:

```
style が Hirazoko または Ichimonji かつ 内径深さ入力あり:
    ドリル深さ = 内径深さ + 0.1

style = YoseRelay:
    ドリル深さ = 対ヨセ長さ + 0.3×ドリル径

workType が ASWD系（J_M8_300/200）:
    ドリル深さ = ASWD_SHOULDER_MM[ドリル径] + CP + 1　（ユーザー入力を上書き）
```

## 5. 実施したUI改修（gui-v2.js）

1. **固定/選択の出し分け**: `classifyDrillMode(workType)` が `"g1"` / `"step"` / `"selectable"` を返し、固定対象のワーク種別・加工方法を選んでいる間は、ドリルモードのG74/G1選択カードの代わりに「固定」バッジ付きの読み取り専用表示を出す。
2. **M12 HGDR固定化のバグ修正**: `classifyDrillMode()` が `resolveM12Profile().finishType` を見て、`"hss"`→ステップ固定・`"baito"`→選択可能・それ以外（HGDR）→G1固定、の3分岐に対応。以前はHGDR選択時が誤って「選択可能」扱いだった。
3. **画面内の表示順序修正**: `q-depths`画面で、ドリル種類の選択カード（HSS/HGDR、M12交差穴の加工方法、G18交差穴の加工方法）を、ドリルモード表示（固定/選択）より**前**に表示するよう並び替え。「原因（どの工具を使うか）→結果（固定か選択か）」の順にすることで、選んだ直後に画面の上に戻って確認する不自然な流れを解消。
4. **刃長制限の警告表示**: `getDrillMaxDepthMM(workType)` で上限値を取得し、ドリル深さがそれを超える場合は次へ進む際に`window.alert`で警告（進行はブロックしない）。

表示文言（「G1（単動）固定」「10mmステップ送り固定」）は暫定。現場で使う言葉に合わせて後で調整する。
